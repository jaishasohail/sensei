1. Project Purpose & Design Philosophy

SENSEI is a mobile accessibility application built for visually impaired users. It combines computer vision, GPS navigation, voice control, spatial audio, and emergency safety into one hands-free experience.

Core design goals:
  • Hands-free operation — Wake word ("Hey Sensei") + 100+ voice command variations
  • Real-world obstacle awareness — On-device object detection (COCO-SSD) with hazard scoring
  • Ground safety — Depth grid analysis for stairs, drop-offs, uneven surfaces
  • Navigation — Outdoor (Google Maps) + indoor (graph/Dijkstra) + pinned locations
  • Hybrid cloud/offline — Heavy AI can run on-device; OCR and logging use LAN server when available
  • Multi-modal feedback — TTS, spatial audio, vibration, on-screen AR overlays

The app is formed as a service-oriented React Native architecture: each capability (detection, navigation, voice, emergency) lives in a singleton service class. Screens orchestrate services; services rarely depend on UI. The backend stores user data, sessions, and serves ML model weights + server-side OCR.

4. Core Algorithms & Logic

4.1 Object Detection Pipeline (COCO-SSD)

File: ObjectDetectionService.js

Model: COCO-SSD (Single Shot MultiBox Detector + MobileNet backbone). Detects 80 object classes (person, car, chair, etc.).

Load sequence

  1. GPU compatibility probe (Samsung Exynos → unknown; MediaTek PowerVR → unsafe/CPU-only)
  2. TensorFlow backend selection: rn-webgl → webgl → cpu, with matMul + depthwiseConv2d validation
  3. Backend result cached in AsyncStorage to skip validation on next cold start
  4. Model load order: device RNFS cache → LAN server (if weights verified) → Google CDN
  5. Non-blocking warmup inference to compile GPU shaders

Per-frame detection (detectFromTensor)

  1. Rate limiting — Target 24 FPS (GPU) or 4 FPS (CPU); skip if previous frame still processing
  2. Tensor prep — Cast to int32 RGB 0–255 as required by coco-ssd
  3. model.detect(tensor, maxBoxes, minScore) — Raw SSD inference
  4. Post-filter — Clamp bbox to image bounds; reject tiny boxes (area thresholds differ for large vs small objects)
  5. False-positive class filter — Food/sports classes require higher confidence (≥0.60)
  6. NMS (Non-Max Suppression) — Per-class greedy NMS in JS (≤8 boxes) or TF async NMS for larger sets. Removes overlapping duplicate boxes.
  7. Distance estimation — Pinhole camera model: distance = (canonicalHeight × focalLength) / bboxHeightPixels. Fallback: 6 / sqrt(bboxArea)
  8. Multi-object tracking — IoU-based association across frames; exponential smoothing on bbox/score/distance
  9. Temporal confirmation — Detections below bypass threshold (0.55) need 2 consecutive frames (minConfirmFrames)
  10. Hazard enrichment — HazardScoringService assigns critical/high/medium/low

Key formulas

focalLength = (frameHeight / 2) / tan(vFOV / 2)
distance    = canonicalHeight × focalLength / bboxHeightPixels
IoU(A,B)      = intersectionArea / unionArea
smoothedBox   = α × newBox + (1-α) × oldBox   (α = smoothingFactor, default 0.72)

4.2 Hazard Scoring

File: HazardScoringService.js

score = baseClassWeight + distanceFactor + sizeFactor + positionFactor

baseClassWeight: car/bus/truck=5, person/dog=3, chair=1, etc.
distanceFactor:  6 - min(5.9, log2(1 + 8/dist) × 2)   → closer = higher
sizeFactor:      min(5, log10(1 + area×100) × 4)
positionFactor:  center=3, left/right=2

Levels: score ≥8 → critical, ≥6 → high, ≥4 → medium, else low

4.3 Depth Estimation (7×5 Ground Grid)

File: DepthEstimationService.js

On-device depth without MiDaS neural network — uses geometry + detection fusion.

  • Camera modeled at ~1 m height; horizon at normalized Y = 0.42
  • Ground distance from bottom-of-frame: depth ≈ GROUND_K / (1 - normY) where GROUND_K = 0.55
  • 7 columns × 5 rows grid maps walkable zones: critical (<0.5 m), near (<1.2 m), mid (<3 m)
  • COCO object heights (person=1.7 m, car=1.45 m) refine per-cell depth via bbox height
  • Stair detection from brightness: ARScreen extracts per-row mean brightness from bottom 60% of frame; horizontal edges (step risers) detected when row-to-row brightness jumps exceed threshold
  • Drop-off detection when depth delta between rows exceeds DROP_THRESHOLD (0.45 m)

4.4 Foot Placement & Safety Guidance

File: FootPlacementService.js

  • Grid-based guidance (primary): Scans 7×5 depth grid columns [left|center|right]; finds safest walk direction
  • Object-based fallback: Uses COCO detections in lower half of frame as ground obstacles
  • Stair temporal filter: Requires 3 consecutive frames before announcing stairs (reduces texture false positives)
  • Cooldown deduplication: Separate TTS cooldowns for guidance (2.5 s), drops (2.5 s), stairs (3 s), surface (4 s)
  • Outputs: Vibration patterns by hazard level + spatial audio beeps + TTS messages

4.5 Voice Command Recognition

File: VoiceCommandService.js (~1,900 lines)

  • Wake word: Continuous listening via expo-speech-recognition; "Hey Sensei" activates command mode
  • Fuzzy matching: Levenshtein distance + similarity score (0–1); keyword containment with 0.8 word similarity threshold
  • 100+ command variations mapped to actions: navigation, AR start/stop, OCR, emergency, indoor nav, pinned locations, emotion detection
  • Command categories: app navigation, AR control, object queries, GPS navigation, indoor navigation, emergency, settings
  • Commands logged to server POST /api/ai/voice-command when authenticated
Levenshtein(a,b): dynamic programming edit distance
similarity(a,b) = 1 - distance / max(len(a), len(b))

4.6 Outdoor Navigation

Files: GoogleMapsService.js, NavigationService.js

  • Route source: Google Directions API (walking mode) with polyline decoding
  • Fallback geocoding: Nominatim/OpenStreetMap when Google API key missing
  • Step tracking: Compare user GPS to next maneuver point; advance step when within threshold (10 m)
  • Guidance: TTS speaks turn instructions; SpatialAudioService plays directional cues
  • Haversine formula for distance between GPS coordinates
  • Bearing calculation for heading-relative turn instructions

4.7 Indoor Navigation (Graph + Dijkstra)

File: IndoorNavigationService.js

Buildings modeled as graphs (not pixel grids) — voice-definable landmarks.

Building → floors → { nodes, edges }
Node:  id, name, type, lat/lng, floor, connectsFloors (stairs/elevator)
Edge:  from, to, distanceM, heading°, instruction text

Pathfinding: Dijkstra's algorithm on weighted edges (distanceM)
Multi-floor: routes traverse staircase/elevator nodes
Localization: nearest node to GPS OR voice "I am at [landmark]"
Storage: AsyncStorage key @sensei_indoor_buildings

4.8 AR Screen Capture Pipeline

File: ARScreen.js

  1. takePictureAsync snapshot (quality 0.55, no base64 in capture)
  2. ImageManipulator native resize to 320 px width + EXIF orientation fix
  3. decodeJpeg (tfjs-react-native) → RGB tensor
  4. Optional resizeBilinear if dimensions off by >2 px
  5. ObjectDetectionService.detectFromTensor
  6. extractImageEdgeSignal — row brightness means for stair detection
  7. Depth + FootPlacement every other frame (performance optimization)
  8. Pipelining: Capture continues while inference runs; latest-frame-wins via pendingPhotoRef

4.9 Emotion Detection

File: EmotionDetectionService.js

  • TinyFaceDetector locates face in JPEG (base64 from front camera)
  • FaceExpressionNet classifies: happy, sad, angry, fearful, disgusted, surprised, neutral
  • Models loaded from LAN /models/face-api/ via fetch + TF.js
  • face-api environment manually configured for React Native (no DOM)

4.10 Emergency & Fall Detection

File: EmergencyService.js

  • Accelerometer magnitude spike detection for falls
  • Manual SOS triggers alert to server + contacts via SMS/WhatsApp
  • Live location broadcast during emergency
  • Socket.IO on server for real-time alert propagation

6. Mobile App — Services Module by Module

ApiService.js

Formed as: Central HTTP gateway for all backend communication.

Logic: Stores JWT in AsyncStorage; attaches Bearer header; clears token on 401. Methods for auth, profile, emergency contacts, navigation sessions, OCR, logs, offline maps.

AppInitializer.js

Formed as: Single boot orchestrator called once after login.

Logic: Parallel model preload + sequential service init; announces online/offline via TTS based on server ping.

ObjectDetectionService.js

Formed as: Core vision engine — largest and most complex service (~1,300 lines).

Logic: Full COCO-SSD pipeline described in Section 4.1. Integrates HazardScoring, voice announcements with per-class cooldowns, server session logging.

DepthEstimationService.js

Formed as: Lightweight on-device alternative to server MiDaS endpoint.

Logic: 7×5 grid depth map from perspective geometry + COCO heights + brightness edge analysis. See Section 4.3.

FootPlacementService.js

Formed as: Safety layer on top of depth + detections for blind navigation.

Logic: Grid/object guidance, stair/drop/surface warnings with temporal filters. See Section 4.4.

HazardScoringService.js

Formed as: Pure scoring function — no side effects, easy to test.

Logic: Weighted sum of class, distance, size, position → 0–10 score → level. See Section 4.2.

VoiceCommandService.js

Formed as: App-wide voice OS — grew to 100+ command patterns.

Logic: Wake word → fuzzy match → action dispatch → navigation ref / service callbacks. See Section 4.5.

GoogleMapsService.js + NavigationService.js

Formed as: Split between API calls (GoogleMaps) and session orchestration (Navigation).

Logic: Route fetch, polyline decode, step tracking, TTS guidance. See Section 4.6.

IndoorNavigationService.js

Formed as: Voice-first indoor alternative when GPS is too coarse.

Logic: Graph storage, Dijkstra pathfinding, landmark CRUD via voice. See Section 4.7.

EmotionDetectionService.js

Formed as: On-device face-api wrapper to avoid server round-trip for emotions.

Logic: TF.js init → load TinyFaceDetector + FaceExpressionNet → detectFromBase64. See Section 4.9.

OCRService.js

Formed as: Server-dependent text reading (signs, labels).

Logic: Health check → POST base64 JPEG to /api/ai/ocr → Tesseract.js on server → confidence-filtered text.

TextToSpeechService.js

Formed as: Central speech output queue to prevent overlapping announcements.

Logic: Priority queue (critical interrupts normal); deduplication; Android callback watchdog.

SpatialAudioService.js

Formed as: 3D audio feedback for obstacle direction and navigation.

Logic: Pan/volume by angle and distance; directional beeps for hazards.

EmergencyService.js

Formed as: Safety-critical module with local + server sync.

Logic: Fall detection, SOS, contact notification, location sharing. See Section 4.10.

OfflineModeService.js

Formed as: Connectivity awareness layer.

Logic: 5 s health ping; toggles cloud mode; caches routes for offline replay; syncs pending data when back online.

10. Data Flow Diagrams

10.1 AR Detection Frame Flow

Camera snapshot → ImageManipulator (320px JPEG) → decodeJpeg → TF tensor → ObjectDetectionService.detectFromTensor → COCO-SSD → NMS → tracking → hazard score → extractImageEdgeSignal (brightness rows) → DepthEstimationService.processFrame → FootPlacementService.processFrame → UI overlays + TTS + SpatialAudio + server log

10.2 Voice Command Flow

Microphone → expo-speech-recognition → "Hey Sensei" wake word detected → Command mode (listening) → normalizeInput + fuzzy match COMMAND_DEFINITIONS → dispatch action (navigate / start AR / SOS / read text / ...) → TextToSpeechService feedback

10.3 Navigation Flow

User: "Navigate to [destination]" → VoiceCommandService → PinnedLocationService fuzzy lookup → GoogleMapsService.getDirections(origin, dest) → NavigationService.startNavigation(route) → LocationService.watchPosition → step advance → TTS turn instructions + SpatialAudio cues → Optional: log session to server
